# 脅威モデル(shibaita CLI)

## 信頼境界

1. **ユーザーのローカル環境**: 改ざん可能。信頼しない。
2. **CLI(このリポジトリ)**: OSSだが利用者が改造して実行できる。信頼しない。
3. **Cloudflare上のサーバー**(`shibaita/web`、非公開): 運営管理。
4. **Google**(外部IdP、Webログイン用): CLI自体はGoogleログインに関与しない。

## 脅威と対策

| 脅威 | 影響 | 対策(MVP) | 残存リスク |
|---|---|---|---|
| ローカルログの改ざん・偽データ生成 | 虚偽のランキング掲載 | **構造的には防がない**(設計判断)。サーバー側のハードキャップ・統計的flag、「参考値」表記、「Verified」等の語を使わない | 巧妙な偽装は載る可能性がある。ネタサービスとして許容 |
| CLI改造によるAPI直叩き・スパム送信 | 不正投稿・スパム | device_token必須、サーバー側でのレート制限(token単位・IP単位)、数値上限、送信スキーマのstrict検証 | 同上。CLIはOSSのため模倣自体は防げない |
| リプレイ・再送 | 二重計上 | サーバー側の置換型UPSERTにより構造的に無害化 | なし |
| ペアリングコードの総当たり | 意図しないデバイス接続 | コード自体はサーバー側で40bitエントロピー+10分TTL+5回失敗で無効化+IPレート制限(サーバー実装側) | 実質不可能 |
| 認証情報(OAuthトークン等)の漏えい | アカウント乗っ取り等 | 本CLIは `~/.claude` 配下の credentials 系ファイルに一切アクセスしない設計。読むのはJSONLの usage フィールドのみ | CLI外の要因(端末侵害等)は対象外 |
| ネットワークコードの混入によるデータ漏えい | 想定外の送信 | `packages/core` / `packages/schema` にネットワークコードが存在しないことをテストで保証。`fetch` 使用は `packages/cli/src/api.ts` のみに限定 | コードレビュー・CIをすり抜けた場合のリスクはゼロではない |
| 送信内容の誤解・過剰送信 | プライバシー侵害の懸念 | allowlist型シリアライズ(DailyUsageから明示的にフィールドを1つずつ写す)+送信直前の公開zodスキーマによるstrict検証。未知キーがあれば送信中止 | なし(スキーマ外キーは構造的に送信不可) |
| 自動・バックグラウンド送信 | 意図しない送信 | 実装しない(設計判断)。送信は `submit` コマンドの明示実行+確認プロンプトのみ | なし |

## 対象外(このCLIのスコープ外)

- Cloudflare上のサーバー実装(`shibaita/web`、非公開リポジトリ)における不正対策の詳細。
- Google OAuthによるWebログインフロー(CLIはこれに関与しない)。
- 端末そのものの侵害(マルウェア等によるログ改ざん)。

## 設計上の原則

- ネットワーク通信コードを1ファイル(`packages/cli/src/api.ts`)に限定し、監査範囲を最小化する。
- 送信は常にユーザーの明示実行時のみ。自動送信・常駐は行わない。
- 「防げない不正」については隠さず明記し、「Verified」「改ざん不可能」等の誤解を招く表現は使わない。
